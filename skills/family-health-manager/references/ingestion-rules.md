# Ingestion Rules

## 适用场景

当用户要求以下任务时，使用本规则：
- 归档新上传医疗资料
- 整理 `inbox/` 中待处理资料
- 为既有事件补充材料
- 更新 `summary.md`、tracker、`profile.md`

## 总流程

1. 确认目标成员
2. 识别日期、机构、科室、事件主题、文件类型
3. 先读取现有档案
4. 判断是新事件、补充材料还是重复材料
5. 创建或复用事件目录
6. 重命名文件
7. 更新 `summary.md`
8. 更新 tracker
9. 仅在必要时更新 `profile.md`
10. 做最终检查

## 写入前先读现有档案

在写任何内容前，至少读取：
- `profile.md`
- `timeline`
- 同日期或同主题附近的 `records/`
- 如涉及药物，读取 `medications`
- 如涉及检查，读取 `measurements`

不要在完全不看现有档案的情况下直接新建目录或追加表格行。

如果任务涉及以下任一情况，先额外读取 `references/deduplication-rules.md`：
- 综合体检
- 综合报告与单项报告并存
- 同机构、近日期、主题相似的多份材料
- 已有事件目录内疑似重复文件

## 判断是否属于既有事件

优先把新资料视为既有事件的补充材料，只要以下因素大体一致：
- 成员一致
- 日期一致或接近
- 医院 / 机构一致
- 科室一致或属于同一就医过程
- 主题一致
- 文件内容明显属于同一次门诊、住院、体检或复查

如属于既有事件：
- 复用原目录
- 更新 `summary.md`
- 仅按需更新 tracker
- 不重复新增 `timeline` 行

## 文件命名

以后所有文件和文件夹都优先使用中文命名。

原始材料命名规则：
- 优先使用报告中最明确、最常见的检查项目名作为文件名
- 如果一份报告只包含单一项目，直接使用该项目名，例如：
  - `血常规.pdf`
  - `甲功五项.pdf`
  - `维生素D.pdf`
  - `HPV分型检测.pdf`
- 如果一份报告包含多种检查项目，使用“最常见的 1 到 2 个项目 + 等多项检查”的形式命名，例如：
  - `血脂四项+肾功能五项等多项检查.pdf`
  - `肿瘤标志物+营养相关指标等多项检查.pdf`
- 影像、病理、门诊记录、出院小结等也优先使用中文且贴近内容的名称，例如：
  - `胸部CT.pdf`
  - `甲状腺超声.pdf`
  - `门诊病历.pdf`
  - `出院小结.pdf`
  - `报告截图.jpg`

命名原则：
- 优先让人一眼看懂这是什么检查或什么材料
- 名称尽量短，但不能失去辨识度
- 不保留无意义的默认文件名，如 `report.pdf`、`IMG_1234.JPG`
- 若同一事件下存在同名材料，可在中文名后补序号或补充说明
- 如果一份材料同时覆盖两个明确项目，使用 `项目A+项目B` 的形式命名，例如 `CT+QCT报告截图.jpg`

不要覆盖已有不同内容的文件。

## 更新 `summary.md`

每个事件目录都应包含 `summary.md`。

更新时：
- 在 `# 摘要` 中概括本次事件
- 在 `# 关键信息` 中写最重要结论或后续安排
- 在 `## 文件说明` 中逐个说明文件
- 对 OCR 不清、日期不明、剂量不明等内容，放入 `## 待确认事项`

不要把原始报告全文复制进 `summary.md`。

如果新资料属于既有事件的补充材料，补充归档后应重新检查：
- `# 摘要` 是否需要补充新的结论或上下文
- `# 关键信息` 是否需要反映新增异常项、阴阳性结果或后续建议
- `## 文件说明` 是否已经覆盖新加入的材料

只有当补充材料明显改变了对事件的理解时，才更新 `timeline` 摘要。

## 更新 tracker

### `medications`
当资料包含处方、医嘱、带药说明或明确用药调整时，更新 `medications`。

写入或去重时，优先使用以下候选键检索既有记录：
- `药品名称 + 开始日期 + 开药机构`

补充规则：
- `状态` 只填写互斥值：`使用中` / `已停用` / `按需` / `暂停` / `不确定`
- “长期用药”不是单独状态；如需表达，写入 `备注`，并在必要时同步到 `profile.md`

### `measurements`
当资料包含化验单、影像报告、体检结果等结构化检查信息时，更新 `measurements`。

写入或去重时，优先使用以下候选键检索既有记录：
- `检查日期 + 医院机构 + 检查大项 + 检查细项`

如果资料只足以确认归档归属，但不足以可靠提取结构化字段，可以只更新 `summary.md`，暂不更新 tracker。

`measurements` 是检查项目级事实和时间序列的主数据源。
不要为了让某项检查可检索，而把同一事件拆成多条 `timeline` 记录。

写入时应同时判断 `状态提示`：
- 优先使用报告原文已有的异常 / 正常标记
- 若报告给出可用参考范围且结果清晰，按结果与参考范围关系填写 `正常`、`偏低` 或 `偏高`
- 无法可靠判断时，才填写 `未标注` 或 `不确定`

对于数值型项目，写入时应优先核对：
- 检测结果
- 单位
- 参考范围
- 状态提示
- 具体项目对应的原始来源文件

对于定性项目，写入时应优先核对：
- 结果结论
- 阴性 / 阳性 / 正常 / 异常等状态
- 来源文件与上下文是否一致

写入 `检查细项` 时：
- 命名规则见 `references/data-model.md` measurements 一节

填写 `来源文件` 时：
- 填写规则见 `references/data-model.md` 来源文件一节

### `timeline`
每个临床相关事件通常只写一行。
补充材料通常更新原事件，不新增第二行。

写入或去重时，优先使用以下候选键检索既有记录：
- `日期 + 医院机构 + 科室 + 事件类型`

仅当补充材料明显改变事件理解时，才更新 `timeline` 中该事件的摘要，例如：
- 新增异常结果
- 新增关键阴阳性结论
- 新增重要影像或病理发现

## 语义去重与保守写回

在追加任何结构化记录前：

1. 先根据成员、日期、机构、主题找候选记录
2. 再结合上下文做语义判断：
   - 是否同一事件
   - 是否同一药物
   - 是否同一检查项
3. 只有在高置信度下才更新已有记录
4. 如果只是补全空字段，可以补全
5. 如果存在冲突或置信度不高：
   - 不自动合并
   - 在 `备注` 或 `summary.md` 中说明
   - 必要时请求用户确认

仅把高置信度字段写入正式结构化记录；低置信度内容统一放入 `## 待确认事项` 或 tracker 的 `备注`。

如果发现同一事件目录内存在疑似重复文件：
- 先核对文件内容是否明显重复
- 能确认重复时，优先收敛处理并在 `summary.md` 中说明
- 不能确认时，再标记为待确认

不要让重复文件长期存在却没有说明。

## OCR 与不确定信息

对图片、扫描件、模糊 PDF、低置信度 OCR，使用保守策略：

- 能确认归属时，先归档原文件
- 不能确认成员时，先放入 `/Family Health Data/inbox/to-confirm-member/`
- 不能确认日期时，先放入 `/Family Health Data/inbox/to-confirm-date/`
- 存在重复嫌疑但暂时无法判定时，先放入 `/Family Health Data/inbox/pending-dedup/`，或在目标事件 `summary.md` 中标记待确认
- 不能确认日期、药名、剂量、关键结果时，不写入正式结构化字段
- 在 `summary.md` 中明确标注待确认项

不要把 OCR 猜测写成确定事实。

## 多页报告

对于多页 PDF 或多页图像材料，不要默认只读取第一页。

应至少确认：
- 是否存在后续结果页
- 是否存在结论页或汇总页
- 是否有第一页未出现的重要项目或异常项

如果当前只完成了部分页的提取：
- 可以先归档
- 可以先录入已确认项目
- 但应在 `summary.md` 或 `备注` 中说明当前结论范围

## `profile.md` 更新门槛

更新门槛见 `references/data-model.md` profile.md 一节。普通单次门诊、轻微异常或未确认信息，通常不更新。

## 归档后的联动更新要求

完成文件归档、移动、重命名，或将资料归入某个既有事件目录后，必须依次评估并执行以下更新：

1. 更新该事件目录下的 `summary.md`
   - 补充新增文件
   - 补充新增事实、关键结论、待确认事项
   - 如新资料只是补充材料，也要检查是否需要修订既有摘要

2. 按资料类型更新 tracker
   - 有用药、处方、停药、换药、剂量变化时，更新 `medications`
   - 有化验、影像、体检、筛查等结构化检查信息时，更新 `measurements`
   - 仅在事件层面的理解发生变化时，才更新 `timeline`

3. 检查是否触发 `profile.md` 更新门槛
   - 新确认过敏或不耐受
   - 重要既往史首次明确
   - 长期疾病背景出现高价值变化
   - 长期稳定用药背景发生高价值变化

不得只完成文件归档而跳过结构化层维护。
如果信息置信度不足，可暂缓更新 tracker 或 `profile.md`，但仍应：
- 先归档原文件到合适位置
- 在 `summary.md` 中写明待确认事项
- 在回复中明确哪些结构化更新已完成、哪些因置信度不足而暂缓

## 完成前检查

结束前检查：
- 成员是否正确
- 事件是否归入正确目录
- 是否误建重复事件
- `summary.md` 是否忠实反映原件
- tracker 是否重复追加
- `来源文件` 是否指向最直接的单项原始文件
- 不确定信息是否被误写成事实
